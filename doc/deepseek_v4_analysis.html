DeepSeek-V4-Pro — 系统级理论建模分析

面向算子 / kernel / 系统三层的工程化建模；对照 DeepSeek-V3.2 基线； 标注：新增 删除/替换 推测

0 · 执行摘要（TL;DR）

  • 体量：总参 1.6T / 激活 49B，61 层，hidden=7168，128 heads；上下文 1 048 576（1M tokens）。
  • Attention：MLA → Hybrid = CSA + HCA 交错堆叠。压缩率由 compress_ratios[61] 逐层指定（大多为 128，少数为 4，末层 0）。CSA 走 index_topk=1024 检索，HCA 走全局稠密但键很短。结果：1M 上下文下，单 token 推理 FLOPs = V3.2 的 27 %，KV Cache = 10 %。
  • MoE：256 × top-8 → 384 × top-6 + 1 shared；scoring 从 sigmoid 换成 sqrtsoftplus；权重 FP4，激活 FP8（e4m3 / ue8m0, block=128×128）。
  • Fusion（最重要的一条）：DeepGEMM Mega-MoE 把 dispatch → linear1 → SwiGLU → linear2 → combine 合并为 单个 mega-kernel，NVLink 通信与 TensorCore 计算在同一 CTA 内 overlap。
  • 残差：mHC（Manifold-Constrained Hyper-Connections） 取代普通 residual，hc_mult=4 路并行残差流，Sinkhorn-Knopp 投影到 Birkhoff polytope 保证双随机矩阵（算子级是一段小 Sinkhorn 循环 + 混合 matmul）。

1.3 关键创新清单

Hybrid Attention (CSA + HCA)

把 KV 序列维度压缩 和 稀疏选择 分离：CSA 走检索（Lightning Indexer → top-1024），HCA 走压缩稠密（每 128 token 压一次）。二者交错，单 token FLOPs ↓73 %、KV ↓90 % vs V3.2@1M。

Lightning Indexer v2

索引器头数 8→64，索引维 128 不变。推测 配合 num_hash_layers=3，走分层哈希召回（LSH → fine-grained score → top-k）。FP4 Indexer 参见 DeepGEMM PR #304。

mHC 残差

hc_mult=4 条并行残差流，混合矩阵用 Sinkhorn-Knopp 迭代投影到 Birkhoff polytope（双随机），谱范数 ≤ 1，深网信号不爆不塌。算子层面：一段 4×4 矩阵上的小循环。

FP8×FP4 混合量化

MoE 专家权重 FP4（e4m3 不适用，应为 mxfp4 或 nvfp4）；激活 FP8 e4m3；scale 用 ue8m0（8-bit 无符号指数，DeepGEMM 约定）。block 128×128。

3.2 通信模式

  • V3.2：经典 all_to_all(dispatch) → expert_fwd → all_to_all(combine)，两次 all-to-all 在 dispatch 模块里串行出现，通信与计算分开 stream。
  • V4 + Mega-MoE：两次 all-to-all 被纳入 mega-kernel 的 producer/consumer 流水（NVLink SM-copy + TensorCore GEMM 同一 CTA 生命周期内重叠）。在 compute_ratio = expert_FLOPs / all_to_all_bytes 降低（top-6、FP4 权重）的前提下，只有 overlap 才能保证 MoE 不重新变成通信-bound。
  • DeepEP v2 layout 推测：和 n_routed_experts=384 的整除友好 layout 相关（384 = 64×6 = 128×3，便于 EP 切分）。

3.3 DeepGEMM 优化思想

  • Grouped GEMM：按 expert 把 tokens 打包，用单个 GEMM kernel 处理多个 [M_i, K] × [K, N] 子问题，分摊 kernel launch 和 SM 分配开销。
  • FP8×FP4 tensorcore 路径：权重 FP4，激活 FP8 e4m3，scale 走 ue8m0 8-bit 指数（块 128×128 细粒度），在 H800/H100 MMA 上用 mma.m16n8k32.f32.e4m3.e2m1 之类指令 推测。
  • Dynamic swap A/B：根据 M/N/K 形状在运行时把 A/B 对调，让 TMA 和 shared memory 路径对齐 swizzle 友好 layout。
  • Symmetric memory：Mega-MoE 要求所有 rank 对 EP buffer 的相对地址一致，NVSHMEM-like 单侧 put/get；否则无法在一个 CTA 里 overlap。

4.1 Attention Fusion

A1. QKV+LoRA 投影融合

V4 改写
before: q_down = matmul(x, W_qa[7168,1536]) q_up = matmul(q_down, W_qb[1536, 128·512]) kv_proj = matmul(x, W_kv[7168, 1·(head_dim + rope)]) q = rmsnorm(q_up); kv = rmsnorm(kv_proj)
▼ fused
after: [q_down, kv_proj] = fused_matmul_split(x, [W_qa | W_kv]) # 单次 HBM 读 x q_up = fused_matmul_rmsnorm(q_down, W_qb) # epilogue 里做 rmsnorm

收益：x 只读一次（7168 维，大头）；Q 的 rmsnorm 写回消除；compute 基本不变，HBM 流量 ≈ 0.5×，decode 阶段直接转 compute-bound。

A2. CSA / HCA 选择性 FlashAttention

新
before: scores = matmul(q, k_compressed.T) # [B, H, Sq, Sk/r] probs = softmax(scores * scale + mask) out = matmul(probs, v_compressed)
▼ fused (FlashAttn2 风格 online-softmax)
after (HCA, ratio=128, dense): out = flash_attn(q, k_c, v_c, scale, window=sliding_window) after (CSA, ratio=4, sparse top-k=1024): idx = lightning_indexer(q, K_hashed) # LSH + score, FP4 权重 out = flash_attn_gather(q, k_c, v_c, idx, scale) # k/v gather fused in-kernel

收益：softmax/matmul/mask 融为单 kernel，无中间 S×S 矩阵落盘；CSA 的 gather 合并到 QK GEMM 的 TMA load，省一次 HBM round-trip。整体 memory ↓ 3–5×（依赖 ratio）。

A3. KV Cache 读写融合（CSA/HCA 差异化）

新
before (每 token decode): k_new, v_new = kv_proj(x) k_cache[t] = k_new; v_cache[t] = v_new # 两次 HBM store out = attn(q, k_cache[:t+1], v_cache[:t+1]) # 一次全量 load
▼ fused
after: HCA 层: compressor(k_new, v_new) 每 r=128 token 触发一次； 压缩 buffer 与 attn kernel 同 CTA，pad 到 128 的余量存 register / smem。 CSA 层: k_new / v_new 经 q_lora 通道先写哈希桶 (num_hash_layers=3)， attention kernel 直接按桶 gather。

收益：HCA 的压缩步骤避免为 127/128 个 token 额外跑 GEMM；CSA 的哈希桶写入让 gather 变成顺序桶内读。估计 KV IO 降至 V3.2 的 10 %（官方数）。

A4. RoPE / RMSNorm 与 MatMul epilogue 融合

V4 多处
before: q = matmul(q_down, W_qb) q = rmsnorm(q) q_rope, q_nope = split(q); q_rope = rope(q_rope, cos, sin) q = concat(q_rope, q_nope)
▼ fused
after: q = matmul_rmsnorm_rope(q_down, W_qb, cos, sin) # epilogue 内联

收益：省去 2 次 HBM 往返（rmsnorm 中间 tensor、rope 中间 tensor）；V3.2 已有，在 V4 下这段代码路径更短（head_dim=512 单通道，无 nope/rope split）。

新 · 与 o_groups=16 相关

4.2 MoE Fusion

M1. Gating + Dispatch 融合

before: logits = matmul(x, W_gate[7168, 384]) scores = sqrtsoftplus(logits + bias_noaux) topk = top6(scores) tokens_by_expert = all_to_all(dispatch(x, topk)) # shuffle
▼ fused
after (single kernel): logits_scores_topk = fused_gate_topk(x, W_gate, bias_noaux) tokens_by_expert = a2a_with_topk_meta(x, topk) # 索引计算在 smem 完成

收益：top-k 排序不落盘；a2a 与 topk 的 index scatter 在同一 warp 完成，省一次 x 的 HBM 读。

M2. Grouped GEMM（DeepGEMM 核心）

before (每专家一个 GEMM kernel): for e in active_experts: h_e = matmul(x_e, W_up[e]) # FP8 × FP8 g_e = silu(matmul(x_e, W_gate[e])) * h_e y_e = matmul(g_e, W_down[e]) # FP8 × FP8
▼ grouped
after (one grouped-gemm kernel): offsets = cum_tokens_per_expert # compile-time layout [H | G] = grouped_matmul_fp8fp4(x_packed, [W_up | W_gate], offsets) g = silu(G) * H Y = grouped_matmul_fp8fp4(g, W_down, offsets)

收益：kernel launch 次数 384→2；SM occupancy 提高（小专家也能吃满 SM）；FP4 权重使得权重 HBM 流量减半。

M3. Mega-MoE（PR #304，V4 旗舰融合）

DeepGEMM v260416
before (5 个独立 kernel，串行): A2A_dispatch → GEMM₁(FP8×FP4) → SwiGLU → GEMM₂(FP8×FP4) → A2A_combine 中间 5 组 HBM 读写，2 次 NVLink round-trip
▼ single mega-kernel
after (producer/consumer SM partitioning): SM-group P : NVLink put/get (dispatch + combine) via symmetric mem SM-group C : grouped-gemm₁ → swiglu → grouped-gemm₂ 两组 CTA 在 ring-like pipeline 中 overlap；中间张量仅驻留 smem/L2。

收益（官方方向，具体数 推测）：

  • HBM 流量：中间 activation（post-gemm1、post-SwiGLU）不落 HBM，↓ 60 %+。
  • NVLink 利用率：dispatch/combine 与 GEMM 完全 overlap，MoE 层 latency 受 max(compute, comm) 而非 compute+comm 支配。
  • Kernel launch：5 → 1，decode 小 batch 时省 microsecond 级 CPU-GPU round-trip。
  • 约束：仅 FP8×FP4，需 PyTorch ≥ 2.9，多进程 + 对称内存。

M4. EP + TP 协同融合（跨切分）

推测
before (TP×EP 解耦): A2A_ep(dispatch) → AllGather_tp(x) → grouped_gemm₁ → AllReduce_tp → … → A2A_ep(combine)
▼ fused topology-aware
after (2D mesh aware): A2A within node (NVLink, EP) ∥ AllGather across node (IB, TP) grouped_gemm₁ 在 token 被 dispatch 进来的 SM 上立刻启动 combine 的 reduce-scatter 折叠进 gemm₂ 的 epilogue

收益：节点内 NVLink / 节点间 IB 两个拓扑层解耦流水；避免 TP allreduce 与 EP all-to-all 互相阻塞。

M5. Shared Expert 融合到主路径

before: y_shared = ffn_shared(x) # 每 token 都跑 y_moe = moe_path(x) # mega-kernel y = y_shared + y_moe
▼ fused
after: y_shared 并入 mega-kernel 的 combine 阶段，累加到 combine reduce-buffer； 不再单独调 ffn_shared，因为 W_shared 已经预 load 到 scoreboard 的固定 SM 组。

收益：shared expert 的 2 次 GEMM 与 MoE gemm₂ 共用同一 epilogue 累加；y_shared 不落 HBM。

6 · 与 xPU-Simulator 本仓库的对齐建议

  • ConfigExtractor：为 model_type == "deepseek_v4" 新增分支，读取 compress_ratios 决定每层是 CSA 还是 HCA；把 num_kv_heads=1 映射到压缩 latent，单独生成 CSA_ATTN / HCA_ATTN OpType。
  • DispatchExtractor：需要为 sqrtsoftplus、o_groups grouped-matmul、mHC 的 Sinkhorn 迭代（小矩阵 20 轮）新增融合规则，否则 ~6 000 aten op 会膨胀。
  • Fusion Rules：core/fusion.py 在 GPU_FUSION_RULES 后追加 MegaMoEFusion（匹配 A2A → GroupedGEMM → SwiGLU → GroupedGEMM → A2A）；NPU 侧用 Ascend910C 的 FA 变体替换 FlashAttention（参考 benchmarks/hf_models/deepseek_v3/ 已有 910C 路径）。
  • KV Cache 模型：core/kv_cache.py 需要支持按层异构压缩（ratio=128 / 4 / 0）、以及 CSA 的 top-k gather 带宽模型（gather 的有效 BW ≈ HBM_BW × top_k / seq_len × gather_efficiency，对 H100 ≈ 0.6–0.8）。
  • Benchmarks：复用 examples/deepseek_v3_2.py 的骨架，复制出 examples/deepseek_v4.py，把 DSA 分支替换成 CSA+HCA 双分支图，并在 1k / 32k / 1M 三点做对比。
